# Question 6

**Describe (with sufficient detail) the TLS (version 1.3) handshake protocol for a client with no certificate and a server using an RSA key (for digital signature only), where elliptic curve ephemeral Diffie-Hellman is used for the key exchange. Explain the mechanisms that are used against possible replay attacks.**

**What are the advantages and drawbacks of using a combination of ephemeral Diffie-Hellman key exchange and RSA digital signature compared to a situation where RSA would be used for both key exchange and authentication (which was possible in TLS version 1.2)?**

## Answer

Handshake protocol: 

client -> server : clienthello 
	- 32 random bit nonce => is for the uniqueness of the session 
	- list of prefered/acceptable cryptographic algorithms, which includes an Hmac based key derivation function. 


client -> server : clienthello-extensions : 
	- suported groups: list of groups for ECDH key exchange 
	- key share: List of DH public keys for client
	- signature algo: list of supported signature algos for later server auth 
	- supported versions 
	
server -> client : ServerHello 
	- 32 random bit nonce 
	- selected cryptographic algorithm + chosen parameters 
	- extensions

server -> client : EncryptedExtensions 
	- encrypted using key derived from server handshake traffic secret 
	- responses to client hello that are not required to determine the cryptographic params of the TLS connection 

Server -> Client: Finished
	 - HMAC over handshake using  with selected Hash function negotiated in ciphersuite and Key derived from server_handshake_traffic_secret ( Different from key used for encrypting this message)


--- from this point on, every message between client and server in encrypted --- 
--- client -> server : encrypted using client_handshake_traffic_secret, which is derived from DH key + handshake so far + nonce ---
--- server -> client : encrypted using server_handshake_traffic_secret, derived from DH key + handshake so far + nonce ---

Each side contributes its own fresh, unpredictable 32-byte value, generated by a secure RNG, and they are independent of each other (p. 23). This guarantees that every handshake is unique, even if everything else were identical (same parties, same certificate, even a reused DH group). An attacker can't simply replay an old recorded handshake, because the new session expects a fresh peer random it has never seen.


*What are the advantages and drawbacks of using a combination of ephemeral Diffie-Hellman key exchange and RSA digital signature compared to a situation where RSA would be used for both key exchange and authentication (which was possible in TLS version 1.2)?*

The benefit of using ephemeral DHE instead of asymmetric key exchange is that we have forward secrecy. This means that even if the private session key gets compromised after a short duration of time this won't be usefull anymore since we generate the keys on the fly.  If we would use RSA in this instance this would not be possible as we would need a fixed private key for the server and if this would get compromised, we lose full data confidentiality.  

Drawbacks of configuration: since no specific RSA is specified i am asuming raw RSA, which is not safe for a number of reasons due to cryptanlysis attacks. Recommended is to use RSA-PSS (PKCS#1-v2.0 scheme) to add structure and salt to make multiplicative properties unuseable
